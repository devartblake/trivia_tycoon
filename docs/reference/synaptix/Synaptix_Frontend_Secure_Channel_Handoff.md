# Synaptix Frontend Secure Channel Implementation Handoff

## Backend Replay/AAD Hardening Handoff - ACTION REQUIRED (2026-05-21)

Repository coordination branch: `docs/backend-secure-channel-handoff-2026-05-21`

Backend status: the Tycoon backend now enforces secure-channel replay metadata and request-context binding on endpoints marked as secure-channel protected. Existing encrypted frontend requests must be updated so every protected mutation sends a fresh request envelope that the backend can bind to the exact request.

### What changed on the backend

- Protected encrypted requests now require all three secure headers:
  - `X-Syn-Sec-Session`
  - `X-Syn-Sec-Seq`
  - `X-Syn-Sec-Nonce`
- `X-Syn-Sec-Seq` must be a positive integer and must match the encrypted request envelope that was generated for that request.
- `X-Syn-Sec-Nonce` must be fresh for each protected request and is used as replay metadata. It should not be reused across retries or different endpoints.
- The backend rejects missing replay metadata, reused sequence/nonce pairs, expired encrypted timestamps, mismatched authenticated subjects, and encrypted bodies bound to the wrong request context.
- Request AAD is bound to the direction, method, full path including query string, session id, sequence number, authenticated subject, and encrypted timestamp.
- Response AAD is generated by the backend using the same request context with response direction.
- Public secure-session negotiation can return either `X25519-HKDF-SHA256-AES256GCM` or the compatibility suite `P256-HKDF-SHA256-AES256GCM`; the frontend must use the actual selected suite reported by the backend.

### Frontend gaps found during local review

- `SecurePayloadCodec` currently binds AAD only to `syn-sec-v1`, method, and `uri.path`; it does not include direction, query string, session id, sequence, subject, or encrypted timestamp.
- `EncryptedApiClient` currently loads the active session after `encryptJson`; because `encryptJson` increments `nextSequence` in storage, the header may be sent with the next sequence rather than the sequence used to create the envelope.
- `EncryptedPayload` does not expose `sequenceNumber`, `replayNonce`, `aad`, or `subjectId` fields to the KMS payload DTO. The backend middleware derives AAD itself, but the frontend must still produce ciphertext using the same AAD string.
- Session start currently advertises only `X25519-HKDF-SHA256-AES256GCM`; frontend session negotiation needs to handle the backend-selected suite, including the P-256 compatibility suite on platforms where X25519 is unavailable.
- Retry behavior must mint a new encrypted envelope and replay nonce. Retrying the same encrypted body should be treated as invalid by design.

### Required frontend implementation behavior

For each encrypted request:

1. Load or start a secure session.
2. Capture the current sequence number before encryption.
3. Generate a fresh replay nonce for this request.
4. Create `encryptedAtUtc` once and use the same timestamp in the encrypted envelope and AAD.
5. Build request AAD with this shape:

```text
syn-sec-v1|request|<METHOD>|<path-and-query>|<session-id-without-dashes>|<sequence>|<subject-id-or-empty>|<encryptedAtUtc-O-format>
```

6. Encrypt the JSON body with the captured request AAD.
7. Send headers using the same captured session id, sequence, and replay nonce.
8. Increment and persist `nextSequence` only after the request envelope and headers are assembled successfully.
9. On retry after session expiry or secure rejection, start a new session if needed and create a brand-new encrypted envelope.
10. Decrypt encrypted responses with response AAD using this shape:

```text
syn-sec-v1|response|<METHOD>|<path-and-query>|<session-id-without-dashes>|<same-sequence>|<subject-id-or-empty>|<same-request-encryptedAtUtc-O-format>
```

### QA checklist for the frontend team

- Verify a protected request succeeds with a fresh session, fresh sequence, fresh replay nonce, and matching request AAD.
- Verify query-string endpoints use the full path plus query string in AAD.
- Verify repeating the same encrypted body is rejected and does not apply the protected action twice.
- Verify retry after session expiry creates a new envelope instead of replaying the failed body.
- Verify sign-out clears secure-session state and sequence counters.
- Verify the frontend uses the backend-selected suite from session start, not a hard-coded suite assumption.
- Verify user-facing errors stay calm and do not expose internal security terms.

See `docs/frontend_backend_secure_channel_sync_2026-05-21.md` for the PR-ready checklist and open coordination items.

## Objective

Add a dedicated secure client-server payload encryption layer to the Flutter app without replacing the existing `ApiService`, `AuthHttpClient`, `SecureStorage`, or auth flow.

This should be implemented as a network security feature, separate from the current local encryption utilities.

## Current frontend fit

The app already initializes these services centrally in `ServiceManager`:

- `ApiService`
- `SecureStorage`
- `EncryptionService`
- `FernetService`
- `AuthTokenStore`
- `AuthHttpClient`
- `SynaptixApiClient`
- SignalR hubs

Add secure-channel support into this same initialization path.

## Data models

### `SecureSession`

```dart
class SecureSession {
  final String sessionId;
  final String protocolVersion;
  final String selectedSuite;
  final List<int> clientToServerKey;
  final List<int> serverToClientKey;
  final DateTime expiresAtUtc;
  final int nextSequence;

  const SecureSession({
    required this.sessionId,
    required this.protocolVersion,
    required this.selectedSuite,
    required this.clientToServerKey,
    required this.serverToClientKey,
    required this.expiresAtUtc,
    required this.nextSequence,
  });

  bool get isExpired => DateTime.now().toUtc().isAfter(expiresAtUtc);
}
```

### `EncryptedPayload`

```dart
class EncryptedPayload {
  final String ciphertext;
  final String nonce;
  final String mac;
  final String contentType;
  final String encryptedAtUtc;

  const EncryptedPayload({
    required this.ciphertext,
    required this.nonce,
    required this.mac,
    required this.contentType,
    required this.encryptedAtUtc,
  });

  Map<String, dynamic> toJson() => {
    'ciphertext': ciphertext,
    'nonce': nonce,
    'mac': mac,
    'contentType': contentType,
    'encryptedAtUtc': encryptedAtUtc,
  };
}
```

## Secure channel flow

### 1. Start session after login

Call:

```http
POST /security/sessions/start
```

Request:

```json
{
  "deviceId": "device-id",
  "clientNonce": "base64url",
  "clientPublicKey": "base64url",
  "supportedSuites": [
    "X25519-HKDF-SHA256-AES256GCM"
  ]
}
```

### 2. Derive directional keys

Use X25519 to derive a shared secret, then HKDF-SHA256 to derive:

- `clientToServerKey`
- `serverToClientKey`

Never store the server root key or any static shared secret in the Flutter app.

### 3. Encrypt selected requests

Wrap only sensitive endpoints first:

- `/auth/login`
- `/auth/refresh`
- `/players/me`
- `/economy/*`
- `/matches/submit`
- `/admin/*` if admin mobile tooling exists
- `/messages/*`
- `/guardians/*`

Do not encrypt public catalog, static asset, or public leaderboard reads at first.

## `EncryptedApiClient` behavior

The wrapper should:

1. Get access token from `AuthTokenStore`.
2. Ensure a secure session exists.
3. Encrypt request body.
4. Add secure headers.
5. Send through existing authenticated HTTP client.
6. Decrypt encrypted responses.
7. Retry once after session renewal if session expired.

Headers:

```http
X-Syn-Sec-Session: <sessionId>
X-Syn-Sec-Seq: <number>
X-Syn-Sec-Nonce: <base64url nonce>
X-Syn-Sec-Version: syn-sec-v1
```

## Storage rules

Use `SecureStorage` only for:

- Current secure session metadata.
- Ephemeral private key if app backgrounding requires it.
- Sequence counter.

Do not store:

- Backend root keys.
- Vault tokens.
- Long-lived symmetric payload keys.
- Hardcoded encryption secrets.

## Testing checklist

- Encrypt/decrypt roundtrip works for JSON payloads.
- Wrong nonce fails decryption.
- Wrong sequence fails on backend.
- Expired secure session renews automatically.
- Logout clears secure session.
- App reinstall invalidates old secure session.
- Web build has fallback behavior if selected crypto is unsupported.
- Performance test with 1 KB, 10 KB, and 100 KB payloads.

## First implementation milestone

Implement `SecureChannelService` and use it only for one non-critical endpoint in dev. Once stable, move to auth refresh, match submission, economy claims, and private messages.
